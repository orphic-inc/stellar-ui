# Contributing to Stellar UI

Welcome! We appreciate your interest in contributing to Stellar UI. To keep the codebase clean, performant, and maintainable, please adhere to the following architectural guidelines.

## Code Standards

### Styling
We have standardized on **TailwindCSS** for styling. 
- Do not add new SCSS or CSS files unless absolutely necessary for complex animations or global reset behavior that Tailwind cannot handle.
- Utilize Tailwind utility classes directly on React components.

### State Management
We use **Redux Toolkit (RTK)**.
- Data fetching and caching should utilize RTK Query where applicable.
- Global application state should be partitioned into clean slices (`createSlice`). 
- Avoid deeply nested or overly complex component-level state if it needs to be accessed by multiple components.

### Type Safety
The UI relies heavily on the OpenAPI schema generated by the `stellar-api` backend.
- Do not manually write API response interfaces.
- Always use `npm run api:generate` to pull the latest types from the API into `src/types/api.ts`.
- Use these generated types throughout your Redux thunks, API queries, and component props.

### Component Structure
- Keep components small and focused on a single responsibility.
- Place reusable visual components in a common UI or components directory.
- Container components (which connect to Redux or API layers) should be kept separate from purely presentational components.

## Submitting Pull Requests
1. Ensure `npm run typecheck` passes without errors.
2. Ensure `npm run test` passes.
3. Ensure Prettier and ESLint run cleanly over your changes.
4. Provide a clear description of the visual or logical changes in your PR and link to any relevant backend API PRs.
